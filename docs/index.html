spillway

replicate Kubernetes Secrets and ConfigMaps across namespaces — via annotations or declarative profiles

Install

helm install spillway oci://ghcr.io/kroy-the-rabbit/charts/spillway \
  --version 0.3.0 \
  --namespace spillway-system \
  --create-namespace

Requires Kubernetes 1.25+ and Helm 3.10+. The chart is published to GHCR as an OCI artifact — no helm repo add needed. The SpillwayProfile CRD is installed automatically.

The idea

You have a Secret in platform and every one of your tenant namespaces needs a copy. Copy-pasting YAML doesn't scale, and writing a script to keep them in sync is exactly as fun as it sounds.

Spillway watches your cluster and handles creation, updates, and cleanup automatically. Annotate a single source object, or use a SpillwayProfile to manage entire sets of objects declaratively — no annotations on the sources required.

Features

Annotation-driven replication

Add one annotation to any Secret or ConfigMap. Replicate to explicit namespaces, glob patterns, or by label selector. Mix and match freely.

SpillwayProfile CRD new in 0.3

Declaratively replicate a set of Secrets and ConfigMaps without touching the source objects. Ideal for platform bootstrap and GitOps workflows.

Key projection new in 0.3

Whitelist or blacklist specific data keys per replica. Give tenants only the keys they need — keep the rest private.

Replica TTL new in 0.3

Set a lifetime on replicas with a Go duration (e.g. 24h). Expired replicas are deleted automatically and recreated with a fresh TTL on the next reconcile.

Namespace consent new in 0.3

Annotate a target namespace to restrict which sources it accepts. Opt in to specific namespaces, specific objects, or leave it open.

Annotation-driven usage

Annotate a source object and spillway does the rest:

apiVersion: v1
kind: Secret
metadata:
  name: shared-api-token
  namespace: platform
  annotations:
    spillway.kroy.io/replicate-to: "team-*,prod"
    spillway.kroy.io/include-keys: "token"   # only copy this key
type: Opaque
stringData:
  token: hunter2
  internal-notes: do not distribute

ConfigMaps work exactly the same way. You can target by namespace label instead of name:

  annotations:
    spillway.kroy.io/replicate-to-matching: "env=prod"
    spillway.kroy.io/exclude-namespaces: "team-dev"
    spillway.kroy.io/replica-ttl: "24h"    # auto-expire after 24 hours

SpillwayProfile

Define replication policy independently of the source objects. No annotations required on the source:

apiVersion: spillway.kroy.io/v1alpha1
kind: SpillwayProfile
metadata:
  name: platform-secrets
  namespace: platform
spec:
  targetNamespaces:
    - "team-*"
  targetSelector:
    matchLabels:
      env: prod
  excludeNamespaces:
    - team-dev
  sources:
    - kind: Secret
      name: shared-api-token
      includeKeys: ["token"]
    - kind: ConfigMap
      name: app-config
      excludeKeys: ["internal-notes"]

The profile reconciler creates and updates replicas as sources change, removes replicas from namespaces that no longer match, and cleans up everything when the profile is deleted.

Namespace consent

Annotate a target namespace to control which sources it will accept:

apiVersion: v1
kind: Namespace
metadata:
  name: payments
  annotations:
    # Accept from the platform namespace only (any object)
    spillway.kroy.io/accept-from: "platform"

    # Or restrict to a specific object
    # spillway.kroy.io/accept-from: "platform:api-token"

Absent annotation means accept from all sources — existing clusters are unaffected by default.

Annotation reference

Annotation Description
replicate-to Comma-separated targets. Accepts all, globs like team-*, or explicit names. Mix freely.
replicate-to-matching Label selector targeting. E.g. env=prod. Resolved at reconcile time.
exclude-namespaces Comma-separated exclusions, same syntax as replicate-to. Always wins over includes.
include-keys Comma-separated data key whitelist. Only listed keys are copied. Mutually exclusive with exclude-keys.
exclude-keys Comma-separated data key blacklist. Listed keys are omitted from replicas.
replica-ttl Go duration (e.g. 24h, 7d). Replicas expire after this duration from their creation and are recreated with a fresh TTL on the next reconcile.
force-adopt Set to "true" to take ownership of a pre-existing unmanaged object instead of skipping it.
accept-from (on Namespace) Comma-separated source allowlist: "platform", "platform:token", "all", or "*".

All annotation keys are prefixed spillway.kroy.io/. kube-system is excluded by default when using all or wildcards — explicitly naming it overrides that.
